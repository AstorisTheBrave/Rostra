## what this is

Audit logging: posts a tidy record to a chosen channel whenever something notable happens — a message is
deleted or edited, a member joins/leaves/is banned, or a role/channel is created or deleted. Staff pick
the channel, which event types to log, and which channels to ignore.

## technical map

**file paths**
- command + module wiring: `src/modules/logging/index.ts`
- event handlers: `src/modules/logging/events.ts`
- config + dispatch: `src/modules/logging/service.ts` (+ `service.test.ts`)
- model: `prisma/schema.prisma` → `LoggingConfig`

**command** — `/logging` (Manage Server): `channel <channel>`, `toggle <type> <bool>`, `status`, and
group `ignore add|remove <channel>`.

**events logged** — messageDelete, messageUpdate, guildMemberAdd/Remove, guildBanAdd/Remove,
roleCreate/Delete, channelCreate/Delete (mapped to the toggles messageDelete, messageEdit, memberJoin,
memberLeave, memberBan, memberUnban, roleChanges, channelChanges).

**exports (service.ts)**
- `getConfig/upsertConfig/invalidate` — cached config
- `sendLog(guild, event, build)` — posts a built container if the channel + toggle are set
- `isIgnored(guildId, channelId)` — ignore-list check
- `truncate(value, max)` — safe string trim for log bodies

**depends on**
- `@/client/defineEvent`, `@/services/database`, `@/utils/components`

## gotchas

- Message delete/edit events require the **Message Content** intent; old (pre-cache) messages arrive as
  partials with no `content`, so the log shows "—".
- Bot-authored messages are skipped; ignored channels short-circuit message logs.
- `sendLog` takes a builder thunk so the container is only constructed when the event is actually enabled.
- Requires `prisma migrate` for the `LoggingConfig` table.
