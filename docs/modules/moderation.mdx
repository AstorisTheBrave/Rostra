## what this is

The moderation feature: server staff use `/mod` to ban, kick, time out, warn, and otherwise discipline
members, and every action is recorded as a numbered "case" so a member's history can be looked up later.
It's the first full feature module and the reference example for how modules are structured.

## technical map

**file paths**
- command + subcommand routing: `src/modules/moderation/index.ts`
- business logic + persistence: `src/modules/moderation/service.ts`
- duration parsing: `src/modules/moderation/duration.ts` (+ `duration.test.ts`)
- database models: `prisma/schema.prisma` → `ModerationCase`, `GuildCaseCounter`

**command** — one top-level `/mod` with subcommands: `ban`, `unban`, `kick`, `timeout`, `untimeout`,
`warn`, `warnings`, `removewarn`, `cases`, `note`, `purge`, `slowmode`, `lock`, `unlock`, `nick`.
Default member permission is `ModerateMembers`; each subcommand additionally checks the specific
permission it needs (e.g. `BanMembers` for ban).

**exports (service.ts)**
- `banUser/unbanUser/kickUser/timeoutUser/removeTimeout/warnUser/addNote(opts): ModResult`
- `createCase(input): Promise<number>` — assigns the next per-guild case number in a transaction
- `getCases(guildId, targetId, type?)` — newest-first history (max 25)
- `deactivateCase(guildId, caseNumber)` — soft-removes a case/warning
- `ModResult` = `{ ok: true; caseNumber? } | { ok: false; messageKey; vars? }`

**depends on**
- `@/services/database` (`getPrisma`) — case persistence
- `@/utils/components` — all replies are Components V2
- `@/i18n` — every message is a `moderation:*` key
- discord.js guild/member APIs for the actual actions

**imported by**
- discovered automatically by the module loader (default export `BotModule`); not imported directly

## gotchas

- Warnings and notes are rows in `ModerationCase` (`type = "warn" | "note"`), not separate tables —
  unifies the history and case numbering.
- Case numbers are per-guild sequential via `GuildCaseCounter.count` incremented inside the same
  transaction as the case insert (safe under concurrency).
- Hierarchy is enforced twice: moderator-vs-target and bot-vs-target top-role position (guild owner
  bypasses the moderator check). discord.js `kickable`/`moderatable`/`bannable` are also checked.
- `removewarn`/`deactivateCase` soft-delete (`active = false`); history is never hard-deleted.
- A `prisma migrate` is required to create the `ModerationCase`/`GuildCaseCounter` tables before these
  commands work against a real database (the client is generated; the migration is applied at deploy).
