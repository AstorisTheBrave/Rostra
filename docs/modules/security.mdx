## what this is

Antinuke: protects a server from a rogue admin or compromised account that tries to mass-delete channels,
mass-ban members, add malicious bots, etc. When a destructive action happens, the bot figures out who
did it (from Discord's audit log), checks whether they're trusted, and if not, punishes them instantly
(ban / kick / strip roles) and alerts the owner. Also includes **anti-raid**: join-flood detection that
locks the server down (raises verification) and auto-lifts.

## technical map

**file paths**
- command + module wiring: `src/modules/security/index.ts`
- detection events: `src/modules/security/events.ts`
- audit-log attribution: `src/modules/security/audit.ts`
- config + punishment: `src/modules/security/service.ts`
- anti-raid lockdown: `src/modules/security/raid.ts` (+ `raid.test.ts`)
- model: `prisma/schema.prisma` → `AntinukeConfig`

**command** - `/security` (Administrator-only): `enable`, `disable`, `status`, `punishment <ban|kick|
strip>`, `antiraid <enabled> [threshold] [window] [minutes]`, `panic <on|off>`, `logchannel <channel>`,
`notify <bool>`, `module <name> <bool>`, and groups `whitelist add|remove` and `extraowner add|remove`.

**protections (gateway events → audit type)** - `antiChannel` (channelCreate/Delete),
`antiRole` (roleCreate/Delete), `antiBan` (guildBanAdd), `antiKick` (guildMemberRemove → MemberKick),
`antiBotAdd` (guildMemberAdd for bots), `antiWebhook` (webhooksUpdate), `antiGuildUpdate` (guildUpdate).

**anti-raid (raid.ts)** - a second `guildMemberAdd` handler tracks an in-memory join window per guild
(`recordJoin`, pure, tested). When `raidThreshold` joins land within `raidWindowSec`, `triggerLockdown`
raises verification to Very High, alerts the log channel/owner, and schedules a durable `raid_unlock`
task (`registerTaskHandler` in index) that restores the prior level after `raidLockMinutes`. `/security
panic on` is a manual lockdown with no auto-lift; `panic off` (or the timer) calls `liftLockdown`. Lock
state + restore level live in an in-memory Map (safe: a guild is single-shard owned); the scheduled
payload carries `{guildId, restoreLevel}` so a restart can still restore correctly.

**exports (service.ts)**
- `getConfig/upsertConfig/invalidate` - cached per-guild config
- `isTrusted({ userId, botId, ownerId, extraOwners, whitelist })` - pure trust check
- `punish(guild, config, userId, reason)` - hierarchy-checked ban/kick/strip + log + owner DM

**depends on**
- `@/client/defineEvent` (events), `@/services/database` (config + cache), `@/utils/components` (logs)

## gotchas

- Detection relies on Discord's **audit log**, which can lag the gateway event slightly - `resolveAudit
  Executor` retries once after 500ms and only matches entries within an 8s TTL.
- An absent `AntinukeConfig` row means antinuke was never set up (treated as disabled). `getConfig`
  caches the row (including the null) per process; `upsertConfig` refreshes the cache.
- Trusted = bot itself, guild owner, extra owners, or whitelisted users - never punished.
- Bot-vs-target role hierarchy is checked before punishing; punishment falls back to stripping roles if
  the primary action fails.
- Antinuke punishes (deterrent) but does not auto-revert deleted channels/roles - a candidate for a later
  pass. Anti-raid now covers the threshold/panic layer.
- Anti-raid raising/restoring verification needs **Manage Server**; failures are logged and swallowed so a
  missing permission never throws on the join path.
- Requires `prisma migrate` to create the `AntinukeConfig` table (now includes the `antiRaid`/`raid*`
  columns).
